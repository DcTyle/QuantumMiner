Identity is the spine of your system:

- `id9 = (x0..x8)`

### 6.2.2 Identity continuity constraint
Between ticks:
- `dist9(id9_next, id9_prev) <= eps_vec9`
unless an explicit identity event occurs.

### 6.2.3 Identity events (must be explicit)
If identity is allowed to change, it must be tagged as one of:
- `MERGE` : two identities combine into one (must define mapping rule)
- `SPLIT` : one identity splits into two (must define children id9 derivation)
- `REKEY` : re‑addressing event with explicit transform `T_id9`

**Rekey transform**
- `id9_next = T_id9(id9_prev, context)`
Where `T_id9` must be an explicit function, e.g.:
- affine transform: `id9_next = A * id9_prev + b`
- or constrained update: only specific axes may move

No implicit identity drift is allowed without an event.

### 6.3.1 `id9_update(state_in) -> state_out`

### 6.3.2 `id9_event_apply(event, state_in) -> state_out`

### 6.3.3 `id9_validate(prev_state, next_state) -> EwId9Report`

### 6.4.1 Fixtures
- **Fixture I: static identity**
 - Identity should not move at all; `dist9 = 0`.
- **Fixture J: continuous identity drift**
 - If allowed: define a small known per‑tick update and verify exactly.
- **Fixture K: merge/split**
 - Provide known mapping and verify determinism.
- **Fixture L: stress under perturbation**
 - Apply bounded perturbations to physics state; identity must remain stable unless explicit event triggers.

### 6.4.2 Pass/fail
- No identity change without an explicit event tag.
- With an event tag, `T_id9` must be deterministic and reproducible.
- Identity continuity must hold within `eps_vec9` for all non‑event ticks.

### 6.4.3 What to do if it fails
- If identity drifts: you have hidden coupling from physics state into id9; either forbid it or make it explicit.
- If merge/split is ambiguous: define canonical ordering (e.g., lexicographic by id9) and document it.
- If rekey is needed: define `T_id9` with axis‑by‑axis rules.
